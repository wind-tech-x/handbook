# コミットルール
## コミット単位
* 今のところ厳密なルールはなし（人数が10人くらいのチームになったとき考える)
* 作業のキリがいいところで

## コミットメッセージ
* 何をやったか具体的に書く
* 実装がIssueと紐づく場合、コミットメッセージにIssue番号を書く
  * 例: #123 ○○の修正
  * #123の後ろには、半角スペースが必要
  * GitHubのIssueのページにコミットが表示されて見やすい

## プルリクエスト

下記のパターンはプルリクエスト必須。みんなでコードレビューしましょう。
* develop -> master
* feature -> develop
* hotfixes -> master
* hotfixes -> develop

また、コードレビューしてもらうためにfeature同士でプルリクエストを投げるのもあり

* feature -> feature
